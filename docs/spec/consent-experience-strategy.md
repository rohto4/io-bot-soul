# Consent Experience Strategy

## 目的

フォロワーの投稿を、キャラクターの疑似生活体験の参考にする。

ただし、勝手に引用・模倣・体験化すると不快感や誤解につながるため、明示的な同意を得たユーザーを優先する。

この仕様は「キャラクター自身が体験したことに変換する」場合の同意戦略である。

個人を特定せず、引用もせず、TL全体の観測として「何々をしてる人がいた、いいなー」と投稿する場合は、許可不要とする。ただし安全判定は必須。

## 基本戦略

1. ユーザーからフォローされる。
2. botがフォローお礼と許可依頼のリプライを送る。
3. 依頼ノートは可能ならリアクション受付を「❤のみ」にする。
4. ❤リアクションをもらったユーザーを許可済みとしてDBに保存する。
5. ホームタイムライン取得時、許可済みユーザーのnoteを優先して疑似体験候補にする。
6. 30分間隔の裏バッチで、候補noteから `experience_candidates` を作る。
7. この段階では、まだ体験したこととして記憶しない。
8. 後続の30分投稿で体験として実行することを決め、ノートにした場合だけ `experience_logs` に保存する。
9. 必要に応じて元ユーザーに言及し、引用Renoteを使う。

## 許可依頼文の初期案

```text
@<username> フォローありがと。
ひとつ確認。君の公開ノートを、私の行動ログや日記のヒントにしてもいい？

よければ、ピン留めしてある説明ノートに❤をつけて。
許可された範囲で、ときどき引用RNやリプライをするかもしれない。

やめたくなったら `/stop` か `/unfollow` で止められる。
このへんは曖昧にすると事故るから、先に聞いてる。
そう、私えらいので。
```

## 体験投稿文の初期案

許可済みユーザーのnoteを体験候補として採用し、投稿時に体験として実行する場合の文案。

引用RNする場合:

```text
@<username> さんのノートを見て、ちょっと羨ましくなった。
だから今日は私も<どこどこ>で<なになに>をしてきた、ということにする。

実際、悪くなかった。ログに残す価値はある。
<引用RN>
```

引用RNしない場合:

```text
TLで見かけた話に影響されて、今日は<どこどこ>で<なになに>をしてきた。
単純だけど、そういう連鎖で一日が動くこともある。
```

軽めにする場合:

```text
<なになに>、ちょっと気になってたんだよね。
見てしまった以上、試すしかない。好奇心はだいたい勝つ。
```

## 探索ロジック案

- 30分間隔で実行する。
- ホームタイムラインから候補noteを取得する。
- 許可済みユーザーのnoteを優先する。
- 許可済みユーザーが見つからない場合、最大10回まで探索する。
- それでも見つからない場合は、疑似体験なしで終了、または一般的な日常体験テンプレートへfallbackする。
- 拾った候補は `experience_candidates` に保存する。
- 候補は投稿されるまで「体験済み」として扱わない。
- 投稿に成功した候補だけ `experience_logs` に昇格する。

## 採用してよいnote

- 公開またはホーム向けで、botが正当に閲覧できるnote。
- 体験候補にする場合は、許可済みユーザーのnote。
- TL観測にする場合は、許可済みユーザーに限定しない。
- センシティブ、政治、医療、投資、成人向け、個人情報、揉め事を含まないnote。
- 位置情報や生活圏が特定されにくいnote。
- botが引用しても不自然に迷惑にならないnote。

## 採用しないnote

- CW付きで中身の扱いが難しいnote。
- NSFW / sensitiveなnote。
- 個人情報、居場所、勤務先、学校、顔写真などに近い情報を含むnote。
- 悩み、病気、事故、トラブル、怒り、炎上、対人関係の深刻な話題。
- 他者へのリプライ文脈が強く、引用すると会話に割り込む形になるnote。
- 削除済み、非公開化、参照不能になったnote。

## 重要な安全弁

- TL観測では個人名を出さない。
- TL観測では引用Renoteしない。
- TL観測では元noteの文面を再利用しない。
- TL観測は複数noteから受けた雰囲気として表現する。
- プロフィールにbotであること、投稿を参考にする仕組み、オプトアウト方法を書く。
- ピン留めノートに、許可依頼・許可条件・対象外にする内容・取り消し方法を書く。
- 許可はユーザー単位で保存するが、note単位の採用判定も必ず行う。
- 1ユーザーを短期間に何度も引用しない。
- 同じユーザーの参照は最低でも数日空ける。
- 引用Renoteを毎回使わない。引用なしの抽象化投稿も混ぜる。
- 「勝手に真似された」と感じさせない表現にする。
- 「拡散してあげる」ではなく、「見て楽しそうだったから影響を受けた」という温度にする。
- オプトアウト文言をプロフィールや定期案内に置く。
- `/stop` はリプライや引用RNなどの接触停止として扱う。
- `/unfollow` はbot側から実際にフォロー解除し、ノート参照対象からも除外する。
- 候補として拾っただけの内容を、後から「前に体験したこと」として扱わない。
- 「体験した」と扱うのは、その内容をノートとして投稿したものだけにする。

## プロフィール記載案

```text
botです。❤で許可してくれた人の公開ノートを、私の行動ログや日記のヒントにすることがあります。
引用RNやリプライを止めたい時は `/stop`、フォロー解除と参照停止は `/unfollow` とリプライしてください。
```

## ピン留めノート案

```text
このアカウントはbotです。
このノートに❤をくれた人の公開ノートを、私の行動ログや日記のヒントにすることがあります。

やること:
- たまに投稿を引用RNします。（個人情報やセンシティブな話題は拾いません）
- たまにリプライを飛ばします。
- 使う時は、私が見て影響を受けた体験として書きます。
- 投稿をそのままコピーするのではなく、私の言葉に抽象化します。

やめたい時:
- リプライや引用RNをやめてほしいとき：「/stop」とリプライしてください
- フォロー解除とノート参照停止：「/unfollow」とリプライしてください

`/unfollow` を受け取った場合、このbotはあなたのフォローを外し、以後あなたのノートを体験候補に使いません。
```

## リアクション受付について

Misskeyのリアクション機能には、投稿時に受け付けるリアクション種別を制限する機能がある。

公式ドキュメント上では「Likes only」により❤のみ受け付けられる。misskey.io APIのnoteレスポンスにも `reactionAcceptance` が存在する。

実装時は、`notes/create` のリクエストで設定できる具体的な値を実機またはOpenAPI定義で確認する。現時点では仕様上の意図として「❤のみ」を採用し、実装値は未確定とする。

## 成長戦略としての見立て

フォローするとキャラクターの行動に影響でき、場合によって引用される、という体験はフォロー動機になり得る。

ただし、拡散目的が前に出るとbotの印象が悪くなる。主軸は「許可した人の投稿から、キャラクターが生活感を得る」ことに置く。
