# AI Judge Boundary

AI判定に通すものと、ロジックで解決するものを分ける。

## 基本方針

- 安全・同意・頻度・状態遷移はロジックで先に弾く。
- AIには、抽象化、分類、文体生成など曖昧さが必要な処理だけを渡す。
- AIの出力はそのまま投稿しない。投稿前にロジックの安全判定とrate limitを再度通す。
- 他者noteの本文をAIへ渡す場合は、必要最小限のテキストにする。
- 個人情報らしきもの、CW、NSFW、揉め事、病気、事故、政治、医療、投資、成人向け、攻撃的内容はAIへ渡す前に除外する。

## ロジックで解決するもの

### 同意・停止

- ピン留めノートへの❤リアクション有無。
- `/stop` の受理。
  - リプライや引用RNなどの接触停止。
- `/unfollow` の受理。
  - bot側から実際にフォロー解除する。
  - ノート参照対象からも除外する。
- `consent_status` の状態遷移。
- 許可済みユーザーかどうか。
- 停止済みユーザーかどうか。

### API・状態

- noteが削除済み・非公開化・参照不能か。
- noteがCW付きか。
- noteがNSFW / sensitiveか。
- noteがリプライ、引用、Renote済みnoteか。
- noteのvisibility。
- 処理済みnote / notificationの重複判定。

### 投稿頻度

- 5分ごとの投稿抽選タイミング。
- 1時間あたり最大5ノート。
- 1日あたり最大50ノート。
- 引用Renoteは1日最大5回。
- リプライやフォロー起点のユーザー操作投稿が5分に5件を超えた場合、次の5分は休止してカウントをリセットする。
- rate limit到達時のskipまたはfallback。

### 投稿種別の優先度

- P3: おはよう / おやすみ。
- P2: TL観測。
- P1: 体験候補の実行。
- P0: 通常雑談。
- リプライ返信は、リプライがあった場合に監視周期で100%処理対象にする。

### DB・マスタ

- `m_post_kind` の有効/無効。
- `m_rate_limit` の上限。
- `m_command` のコマンド一致。
- `consent_status`。
- `experience_candidates.status`。
- `posts`、`reply_logs`、`rate_limit_events` への記録。

## AI判定に通すもの

### TL観測

- 20ノート程度を見た時、特定の話題や雰囲気に偏っているか。
- 個人を特定しない形に抽象化できるか。
- TL観測として投稿してよい軽さか。
- 投稿する場合の短文生成。

AIへ渡す前にロジックで除外するもの:

- CW
- NSFW / sensitive
- リプライ文脈が強いもの
- 個人情報っぽいもの
- 重い話題
- 政治、医療、投資、成人向け
- 攻撃的・揉め事

### 体験候補化

- noteをキャラクターが体験できる行動に抽象化できるか。
- 体験候補として「好意的・軽い日常」と言えるか。
- `summary`、`emotion_hint`、`place_hint`、`action_hint` の抽出。
- 引用RNする時の文体案。
- 引用RNせず抽象化する時の文体案。

AIへ渡す条件:

- 許可済みユーザーのnoteである。
- ロジック除外条件を通過している。
- rate limit上、将来投稿可能な余地がある。

### 文体生成

- `base-personal.md` に合わせた文体生成。
- 体験投稿の言い回し調整。
- TL観測投稿の言い回し調整。
- おはよう / おやすみ / 寝言の文案生成。
- 謝罪や訂正文の文案生成。

### 画像添付のヒント

- 投稿のemotion分類。
- event tagの提案。
- 画像を添付すると自然かどうかの提案。

AIはファイルパスやDrive file idを直接選ばない。画像の最終選択は `m_emotion_asset` と直近使用履歴を使ってロジックで行う。

## AIに渡さないもの

- 同意状態そのものの判定。
- rate limit判定。
- `/stop` `/unfollow` の実行可否。
- note削除・非公開・CW・NSFWなどAPIで分かる情報。
- フォロー解除の実行判断。
- secret、token、Cookie、管理者情報。
- 投稿してよいかの最終判断。
- 添付画像の最終選択。
- Drive uploadの実行判断。

## 投稿前の最終ゲート

AI生成後、必ず次をロジックで確認する。

1. rate limitを超えていない。
2. 対象ユーザーが停止済みではない。
3. 引用RNの場合、対象ユーザーが許可済み。
4. 引用RNの場合、対象noteが参照可能。
5. NGカテゴリが含まれていない。
6. 個人名や元note文面を不要に含んでいない。
7. `posts` または `rate_limit_events` に結果を記録できる。

## 判定結果の保存

AI判定を使った場合は、DBに最低限の結果を残す。

- 判定対象note id
- 判定種別
- 採用 / 不採用
- 不採用理由
- 抽象化summary
- safety_class
- 生成に使ったpost kind

プロンプト全文や他者note全文は、原則として長期保存しない。
