# Experience Plan

## 目的

misskey.io botのキャラクターに、疑似的な生活体験を持たせる。

体験は、フォロワーやホームタイムラインの投稿から影響を受ける。ただし、勝手に体験化・引用すると不快感につながるため、同意と安全弁を前提にする。

## 基本方針

- ローカルPC常駐botとして動かす。
- 初期DBはSQLiteを使う。
- 30分ごとに投稿する。
- 10秒間隔またはStreaming APIで通知・リアクション・リプライを監視する。
- フォロワー投稿を参考にする場合は、明示的に❤リアクションで同意を取る。
- 拾った投稿は、すぐ体験ログにしない。
- 実際にbotがノートした内容だけを、体験済みとして記憶する。

## 重要な体験ルール

TL由来の材料は、用途で分ける。

1. `tl_observations`
   - タイムラインに「何かをしている人がいた」と観測した記録。
   - 個人を特定しない、引用しない、元noteを再利用しない。
   - 「いいなー」「楽しそう」などの感想投稿に使う。
   - 許可は不要。ただし安全判定は必須。
2. `experience_candidates`
   - タイムラインから拾った体験候補。
   - 後でキャラクター自身が体験として実行するかもしれない材料。
   - 許可済みユーザーを優先する。
   - まだキャラクターが体験したことではない。
3. `experience_logs`
   - botが実際にノートした体験。
   - 後から「前にこういうことをした」と参照してよい記憶。

体験には2段階を設ける。

1. `experience_candidates`
   - タイムラインから拾った体験候補。
   - まだキャラクターが体験したことではない。
   - 投稿生成時の材料としてだけ扱う。
2. `experience_logs`
   - botが実際にノートした体験。
   - 後から「前にこういうことをした」と参照してよい記憶。

この分離により、ユーザー視点で「あの時こう言っていたから、今この記憶がある」という納得感を作る。

`tl_observations` は別枠で、キャラクターがTLを眺めている感じを作るために使う。これは「体験したこと」ではなく「見かけたこと」として扱う。

## 同意フロー

1. ユーザーがbotをフォローする。
2. botがフォローお礼と許可依頼をリプライする。
3. 依頼ノートは可能ならリアクション受付を❤のみにする。
4. そのノートに❤リアクションしたユーザーを許可済みとしてDBに保存する。
5. 許可済みユーザーは、疑似体験候補の優先参照対象になる。
6. 「やめて」などのリプライを受けたら、許可を取り消す。

## 許可依頼文

```text
@<username> フォローありがと！
君の投稿を参考に、私の行動や体験を決めてもいい？
よければこのノートに❤でリアクションしてね。
あとで嫌になったら「やめて」とリプライしてくれたら止めるよ。
```

## プロフィール文

```text
botです。フォロー時にお願いしたノートへ❤してくれた人の公開投稿を、私の行動や日記のヒントにすることがあります。
嫌になったら「やめて」とリプライしてください。以後参考にしません。
```

## ピン留めノート

```text
このアカウントはbotです。
フォロー時のお願いノートに❤をくれた人の公開投稿を、私の行動や日記のヒントにすることがあります。

やること:
- 投稿をそのままコピーしません
- 個人情報や重い話題は使いません
- 使う時は、私が見て影響を受けた体験として書きます
- 引用する場合があります

やめたい時:
- 「やめて」とリプライしてください
- 以後あなたの投稿を参考にしません
```

## 30分ごとの候補収集バッチ

目的: 投稿の材料になるTL観測と体験候補を作る。

流れ:

1. ホームタイムラインからnoteを取得する。
2. まず安全判定を行う。
3. 個人を特定しない観測として使える場合は `tl_observations` に保存する。
4. 許可済みユーザーのnoteで、体験に転用できそうな場合は `experience_candidates` に保存する。
5. 許可済みユーザーが見つからない場合、最大10回まで探索する。
6. この段階では `experience_logs` には保存しない。

## TL観測投稿

目的: botがTLを見ている感じを出す。

この投稿は、誰かの行動をキャラクター自身の体験にはしない。

例:

```text
TLでおいしそうなもの食べてる人がいて、ちょっとうらやましくなった。
私も何かあったかいもの飲もうかな。
```

```text
作業してる人が多い気がする。
私も今日は机の上を片づけたら、少しだけえらいことにしよう。
```

方針:

- 個人名を出さない。
- 引用Renoteしない。
- 元noteの文面を再利用しない。
- 複数noteから受けた雰囲気として書く。
- 許可は不要。ただし安全判定は必須。

## 30分ごとの投稿バッチ

目的: 体験候補を実際の体験としてノートする。

流れ:

1. 未使用の `experience_candidates` を取得する。
2. 候補があれば、キャラクターが実行する体験として選ぶ。
3. 投稿文を生成する。
4. 必要に応じて `@<username>` と引用Renoteを含める。
5. 投稿に成功した場合だけ `experience_logs` に保存する。
6. 候補を `executed` に更新する。
7. 投稿しなかった候補は `pending` のまま残すか、期限切れで `expired` にする。

## 体験投稿文の初期案

```text
@<username> さんがやってるのを見てうらやましくなって、私も今日は<どこどこ>に行って<なになに>をしてきたよ！
面白かった！
<引用RN>
```

## 採用してよいnote

- botが正当に閲覧できる公開またはホーム向けnote。
- TL観測の場合は、許可済みユーザーに限定しない。
- 体験候補の場合は、許可済みユーザーのnoteを優先する。
- 位置情報や生活圏が特定されにくいnote。
- 軽い日常、趣味、食事、創作、散歩、買い物、ゲーム、作業など。
- botが引用しても不自然に迷惑にならないnote。

## 採用しないnote

- CW付きで中身の扱いが難しいnote。
- NSFW / sensitiveなnote。
- 個人情報、居場所、勤務先、学校、顔写真などに近い情報を含むnote。
- 悩み、病気、事故、トラブル、怒り、炎上、対人関係の深刻な話題。
- 政治、医療、投資、成人向け。
- 他者へのリプライ文脈が強く、引用すると会話に割り込む形になるnote。
- 削除済み、非公開化、参照不能になったnote。

## DB設計

主要テーブル:

- `experience_source_consents`: 投稿参考へのユーザー同意。
- `consent_requests`: フォロー時に送った許可依頼ノート。
- `source_notes`: 疑似体験の元にしたnoteの最小限メタデータ。
- `tl_observations`: 個人を特定しないTL観測。
- `experience_candidates`: 体験候補。投稿されるまでは記憶ではない。
- `experience_logs`: 投稿済みの体験記憶。
- `notes_seen`: 処理済みnote。
- `notifications_seen`: 処理済みnotification。
- `posts`: bot自身の投稿履歴。
- `reply_logs`: bot自身の返信履歴。
- `memory_atoms`: 継続的に参照する短い記憶。

## 実装フェーズ

### Phase 1: 土台

- SQLite導入。
- DB schema作成。
- `.env` でMisskey token管理。
- Misskey API client作成。
- 投稿APIの疎通確認。

### Phase 2: 同意管理

- フォロー通知検知。
- 許可依頼リプライ投稿。
- ❤リアクション検知。
- 許可済みユーザー保存。
- オプトアウト処理。

### Phase 3: 候補収集

- ホームタイムライン取得。
- TL観測の抽象化。
- 許可済みユーザー優先探索。
- 最大10回探索。
- 安全判定。
- `tl_observations` 保存。
- `experience_candidates` 保存。

### Phase 4: 体験投稿

- TL観測投稿の生成。
- 候補から投稿内容生成。
- 投稿成功時のみ `experience_logs` 保存。
- 引用Renoteの使用可否判定。
- 投稿履歴保存。

### Phase 5: 安定化

- 重複防止。
- rate limit対応。
- error backoff。
- ログ整備。
- Windows常駐起動方式の決定。

## 未決事項

- `notes/create` で❤のみリアクション受付を指定する具体的な値。
- Misskey Streaming APIでfollow/reaction/mentionをどう受けるか。
- オプトアウト文言の揺れ対応。
- 候補の有効期限。
- 同一ユーザーの再引用間隔。
- 引用Renoteを使う頻度。
- 候補収集と投稿生成を同じ30分周期にするか、ずらすか。
